電力データ共有の「信頼の壁」

DSO の悩み

  • 系統混雑を解消するにはDERの柔軟性が必要
  • しかし系統トポロジは企業秘密
  • データを渡したら使い方を制御できない

Aggregator の悩み

  • 最適なDR制御にはフィーダー制約が必要
  • しかし個別DERの状態は競争優位
  • DSOに全デバイス情報を渡したくない

需要家の悩み

  • 電気代削減のためDRに参加したい
  • しかし生活パターンがバレる
  • 同意を撤回しても反映されるか不安

→ 全員の要求を同時に満たす仕組みが必要

データスペースという解決策

データの主権を保持したまま、
信頼された仕組みでデータを交換する

原則1: データは出ない

実データは元のノードに留まる。
共有されるのはメタデータのみ

原則2: 契約が必須

全てのデータ交換に
機械実行可能な契約が必要

原則3: 目的が制限

「混雑管理」で渡したデータは
他の目的に使えない

原則4: 全て記録

全交換をSHA-256ハッシュ付きで
改ざん不可能な形で記録

5層アーキテクチャ

層名称責務実装
5Access / Exchange実際のデータ転送REST API + Kafka
4Policy / Contract / Consentアクセス可否の判定契約状態遷移マシン + ポリシーエンジン
3Catalog / Discoveryデータアセットの発見フェデレーテッドカタログ
2Semantic Modelデータの意味の統一CIM / IEC 61850 / OpenADR
1Identity / Trust認証と信頼の確立Keycloak OIDC + mTLS

各層が独立した責務を持ち、上位層は下位層に依存

3つのデータ主権者

🔴 DSO（配電事業者）

  • 混雑信号を送信

Port 8002

  • 柔軟性エンベロープを提出
  • DR指令を実行
  • 実績を報告

見えないもの: 系統トポロジ

🟠 Prosumer（需要家）

Port 8003

  • 同意に基づきデータを共有
  • 目的別に自動匿名化
  • いつでも同意撤回可能

見えないもの: 隣の家、系統

電柱1本のデータ主権

DSOは柱ごとの集約負荷のみ。Aggregatorは柔軟性のみ。各家庭は自分だけ。

各参加者にとって何が嬉しいのか

DSO（ネットワーク側）

  • 系統トポロジを渡さずに混雑を解消できた
  • 契約で目的を限定、再配布を禁止
  • 緊急時は優先アクセス（監査付き）
  • 全交換のSHA-256監査証跡が残る
  • N社との連携が個別API開発不要

→ 安心してデータを出せるようになった

Aggregator

  • フィーダー制約を取得でき最適なDR制御が可能に
  • 個別DER状態を隠したまま柔軟性を提供
  • 契約でデータ利用の権利を明確化
  • カタログで新しいデータソースを自動発見
  • 複数DSOとの連携がスケーラブル

→ 競争優位を守りながらサービスを拡大

需要家（Prosumer）

  • 生データが外に出ない（技術的に不可能）
  • 目的ごとに自動匿名化される
  • 同意をいつでも即座に撤回可能
  • DR参加で電気代削減の恩恵
  • 隣の家のデータは絶対に見えない

→ 安心してDRに参加できるようになった

Before / After データスペース

BEFORE（従来のAPI連携）

  • データを渡したら制御不能
  • 目的外利用を技術的に防止できない
  • 同意は形式的なチェックボックス
  • N社連携 = N×(N-1)/2 の個別開発
  • 監査は各社バラバラのログ
  • 不正アクセスの事後検知が困難

AFTER（データスペース）

  • 契約で利用条件を機械的に強制
  • 目的外アクセスは自動拒否 + 監査記録
  • 同意撤回が即座に技術的に反映
  • N社がデータスペースに参加するだけ
  • 全交換をSHA-256で改ざん検出
  • 不正アクセスも全て記録・追跡可能
